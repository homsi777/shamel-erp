# PostgreSQL 16 Query And Coupling Risk Register

Assessment date: 2026-04-01

Purpose: inventory concrete SQLite couplings found in the codebase and map each one to PostgreSQL 16 migration risk and remediation direction.

## Rating scale

- `Low`: portable with minimal change.
- `Medium`: requires targeted code adaptation or stronger typing.
- `High`: requires refactor, new abstraction, or data cleanup.
- `Critical`: blocks safe PostgreSQL cutover unless addressed first.

## Cross-cutting conclusions

- The backend server migration is feasible without rewriting the ERP, but it is not a simple driver swap.
- Three areas block a safe direct cutover: schema typing, runtime schema mutation, and SQLite-specific operational tooling.
- The Android local runtime is a separate SQLite subsystem. If strategic direction truly means “fully leave SQLite behind”, that decision needs its own project boundary and execution plan.
