# PostgreSQL 16 Schema Compatibility Matrix

Assessment date: 2026-04-01

## Field-class recommendations

| Field class | Current pattern in repo | Evidence | PostgreSQL 16 recommendation | Status |
|---|---|---|---|---|
| Tenant identifiers | `text` nullable `company_id`, `branch_id` on many tables | `backend/db/schema.ts` across most tables | Keep as `text` if IDs are application-generated strings; make non-null on operational tables where scope is mandatory | Adapt |
| Integer PKs for accounting | `integer primary key autoincrement` / SQLite rowid-backed | `accounts`, `journal_entries`, `journal_entry_lines` in `backend/db/schema.ts` and `0019_fix_accounts_id_type.sql` | Use `generated by default as identity` or `bigint identity`; prefer `bigint` for future safety | Adapt |
| String PKs | `text primary key` | Most operational tables | Keep as `text`/`uuid` depending on existing ID generation strategy; preserve current IDs during migration | Safe with validation |
| Monetary amounts | `real` | invoices, vouchers, journal lines, FX tables, expenses, consignment, payroll | Use `numeric(18,4)` minimum; use tighter scales per domain if known | Dangerous as-is |
| Exchange rates | `real` | `exchange_rate`, FX revaluation tables | Use `numeric(18,8)` or domain-specific scale | Dangerous as-is |
| Quantities / stock | `real` | item qty, base_qty, textile lengths, manufacturing qty | Use `numeric(18,4)` or `numeric(18,6)` based on textile/base-unit precision | Dangerous as-is |
| Booleans | SQLite integer booleans | many `integer(...,{mode:'boolean'})` | Use native `boolean`; normalize imported `0/1/null` | Adapt |
| Business dates | `text` storing `YYYY-MM-DD` or ISO strings | invoices.date, vouchers.date, entry_date, join_date | Use `date` where semantics are calendar/business date | Adapt |
| Audit timestamps | `text default CURRENT_TIMESTAMP` or JS `toISOString()` | `created_at`, `updated_at`, `posted_at`, `resolved_at` | Use `timestamptz`; standardize timezone and write path | Adapt |
| JSON documents | `text` JSON blobs | `items`, `metadata`, `template_json`, `print_meta`, `audit` | Use `jsonb` when queried, filtered, or partially updated; retain `text` only for opaque payload archives | Adapt |
| Status / enum-like values | freeform text | `status`, `type`, `severity`, `payment_type`, `session_status` | Keep as `text` initially, add `check` constraints for stabilized domains later | Adapt |
| Global unique business codes | plain `unique` or app logic only | `accounts.code`, `companies.code`, document numbers, serial numbers | Re-evaluate scope; many should become composite unique `(company_id, branch_id, key)` | Adapt |
| Foreign keys | partial and inconsistent | docs note many logical scoping refs not enforced as SQLite FK | Add FKs incrementally after orphan audit; start with accounting lineage and restaurant request/session lineage | Adapt |

## SQLite vs PostgreSQL behavior matrix

| Topic | Current repo behavior | PostgreSQL 16 behavior | Impact | Action |
|---|---|---|---|---|
| Type coercion | SQLite accepts mixed text/integer comparisons more freely | PostgreSQL is stricter | High | Normalize mismatched columns, especially journal linkage IDs |
| `REAL` precision | Binary float | Binary float unless changed | Critical | Move financial and quantity fields to `numeric` |
| `CURRENT_TIMESTAMP` on text columns | Stores text timestamp | On typed timestamp columns returns proper timestamp value | Medium | Convert column types and defaults together |
| `PRAGMA` | Used for schema and integrity introspection | Not supported | Critical | Replace with `information_schema` / catalog queries |
| `sqlite_master` | Used by scripts | Not supported | Critical | Replace with `pg_catalog` / `information_schema` |
| `rowid` | Used for deterministic ordering and legacy remap | Not available | Critical | Replace with explicit PK/timestamp ordering |
| `AUTOINCREMENT` | SQLite-specific identity semantics | PostgreSQL uses sequences / identity | High | Rebuild integer PK strategy |
| `INSERT OR REPLACE` | Delete+insert flavored upsert semantics | Not supported | Critical | Replace with `INSERT ... ON CONFLICT DO UPDATE` |
| `INSERT OR IGNORE` | Ignore duplicates | Not supported directly | High | Use `ON CONFLICT DO NOTHING` |
| `last_insert_rowid()` | Fetches last rowid | Not supported | Critical | Use `RETURNING` |
| `SUBSTR(date_text,1,10)` filtering | Common in reports | Works but kills typed-date advantages and may differ on null/text behavior | High | Convert to typed date/timestamp comparisons |
| `LIKE` | Case behavior depends on SQLite defaults/collation | Case-sensitive by default; `ILIKE` for case-insensitive | Medium | Audit search/filter paths; use explicit normalization or `ILIKE` |
| Empty string vs null | Often tolerated and treated similarly in app logic | Distinct values | Medium | Normalize import and add explicit null/empty-string handling |
| Foreign keys | Can be present but historically inconsistently enforced | Strong and always validated once enabled | High | Run orphan audit before adding FKs |
| Transactions | Single-writer SQLite, low concurrency | MVCC with row-level locking and different contention patterns | High | Re-test queue counters, invoice posting, provisioning, settlement flows |
| Savepoints / nesting | Existing code contains SQLite workaround comments | PostgreSQL supports savepoints, but deadlock/order behavior differs | Medium | Standardize transaction boundaries and lock ordering |
| Partial indexes | Limited use today | Strong PostgreSQL feature | Opportunity | Add partial indexes for active/open/manual-review workflows after cutover |
| Check constraints | Rare today | First-class | Opportunity | Add validated constraints for status, non-negative amounts, rate ranges after cleanup |

## Table-class recommendations

| Table / class | Current state | Recommendation |
|---|---|---|
| `accounts`, `journal_entries`, `journal_entry_lines` | Core accounting tables with integer IDs, `real` money, partial FKs | Migrate first-class; use `bigint identity`, `numeric`, strict FK model |
| `vouchers` | Mixed-type journal FK, large population missing journal links | Normalize and reconcile before enabling strict FK |
| `invoices` | Text JSON items, money fields, source-document links, printing flags | Keep logical shape; convert totals to `numeric`, timestamps to `timestamptz`, consider `items` as `jsonb` |
| `party_transactions`, `inventory_movements` | Canonical ledgers with text dates and `real` values | Treat as cutover-critical canonical ledgers; strong indexes required |
| `system_events`, `audit_logs`, `print_jobs` | Event/audit tables with JSON text metadata and append-mostly access | Move metadata to `jsonb`, add descending timestamp indexes |
| `restaurant_*` | Newer, better-structured operational tables | Mostly portable; add composite indexes and typed timestamps |
| `system_settings`, `print_templates` | Key/value and template JSON | Use `jsonb` where structured reads matter |
| `textile_*`, `consignment_*` | Domain-specific quantity/price heavy tables | Use `numeric`, add scope and document indexes, validate unit precision |

## Immediate schema blockers

1. `vouchers.journal_entry_id` must be normalized from `text` to integer/bigint.
2. Financial and quantity `real` columns must be retyped before production PostgreSQL use.
3. Runtime schema mutation in `seed-accounts.ts` and `printJobService.ts` must be removed from application execution paths.
4. The PostgreSQL migration history should start from a clean baseline, not from replaying SQLite SQL files.
